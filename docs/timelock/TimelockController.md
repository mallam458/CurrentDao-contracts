# ⏰ Timelock Controller

The **Timelock Controller** is a critical security component of the Current DAO. It acts as a safety buffer between a governance decision being made and its actual execution. This delay (typically 48 hours) provides several key protections:

1.  **Security against malicious proposals**: If a governance decision is captured by malicious actors (e.g., a flashloan attack), the community has time to respond and potentially cancel the action using emergency mechanisms.
2.  **Community Review Period**: Token holders have a final window to review the exact transaction data being executed and can exit the protocol (e.g., withdraw liquidity) if they strongly disagree with the decision.
3.  **Governance Order**: Ensures that multiple governance decisions are executed in the correct order by using the `predecessor` mechanism.

## 🚀 How it Works

Proposals that pass the voting stage are **scheduled** in the Timelock. This puts the operation into a `Pending` state. Once the minimum delay has passed, anyone with the **executor** role can trigger the execution.

### ✅ Access Control Roles

The Timelock uses a granular role-based access control (RBAC) system:

-   **PROPOSER**: Can schedule operations. Usually granted to the Governance contract.
-   **EXECUTOR**: Can trigger the execution of scheduled operations once the delay has passed. This can be the Governance contract or a set of trusted community members.
-   **CANCELLER**: Can cancel a pending operation. This is restricted to the DAO admin or a specific emergency multisig.
-   **ADMIN**: Can manage roles and update the minimum delay.

## ⚙️ Configuration Parameters

-   **Minimum Delay**: The shortest period an operation must wait before execution (Default: 48 hours).
-   **Operation ID**: A unique identifier generated by hashing the target, value, calldata, predecessor, and salt of an operation.

### Emergency Procedures

In the event of a critical bug or a malicious proposal passing, the **CANCELLER** address can nullify the pending operation. This is most effective when combined with the 48-hour delay window.

## Example Lifecycle

1.  **Governance** schedules an operation: `timelock.schedule(...)`
2.  **Delay** of 48 hours passes.
3.  **Executor** triggers execution: `timelock.execute(...)`
4.  **Transaction** is finalized and state changes are applied.
